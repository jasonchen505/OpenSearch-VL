## 增量学习点

### 1. 资源受限下的系统设计思维

#### 1.1 从"理想方案"到"可行方案"的转变

**官方方案（理想）**：
- SFT：256×H20 GPU，全参数微调，序列长度32000
- RL：64×H100 GPU，Megatron TP/PP/CP，序列长度70000

**我们的方案（可行）**：
- SFT：8×3090 GPU，LoRA微调，序列长度8192
- RL：8×3090 GPU，简化并行，序列长度16384

**关键思维转变**：
1. **从全参数到LoRA**：牺牲5-10%性能，换取4×显存节省
2. **从长序列到短序列**：牺牲部分能力，换取4×显存节省
3. **从高并行到低并行**：牺牲速度，换取可行性

**面试应用场景**：
> **Q：如果给你一个需要256张H100的项目，但你只有8张3090，怎么办？**
>
> **A**：首先分析瓶颈在哪里（显存、算力、通信），然后针对性优化：
> 1. 显存瓶颈：使用LoRA/QLoRA、CPU Offload、序列长度缩减
> 2. 算力瓶颈：减少训练epochs、使用更小的模型
> 3. 通信瓶颈：使用单节点、减少并行度
> 关键是在性能损失和可行性之间找到平衡点。

#### 1.2 显存优化的层次化策略

**第一层：算法层面**
- LoRA：只训练0.1%的参数
- QLoRA：4-bit量化 + LoRA
- Gradient Checkpointing：用计算换显存

**第二层：系统层面**
- CPU Offload：将优化器状态移到CPU
- DeepSpeed ZeRO-3：分片模型参数
- 动态批量大小：根据序列长度调整

**第三层：配置层面**
- 减小batch size
- 减小序列长度
- 减少并行度

**面试应用场景**：
> **Q：如何系统性地优化显存使用？**
>
> **A**：采用层次化策略，从算法层面到系统层面再到配置层面：
> 1. 算法层面：LoRA减少可训练参数，Gradient Checkpointing减少激活值
> 2. 系统层面：CPU Offload卸载优化器状态，DeepSpeed分片模型参数
> 3. 配置层面：减小batch size和序列长度
> 每一层都有不同的优化粒度和性能代价，需要综合考虑。

### 2. LoRA微调的深入理解

#### 2.1 LoRA vs 全参数微调

**LoRA原理**：
```
W' = W + ΔW = W + BA
其中 B ∈ R^{d×r}, A ∈ R^{r×d}, r << d
```

**显存对比**：
| 方案 | 可训练参数 | 优化器状态 | 梯度 | 总显存 |
|------|-----------|-----------|------|--------|
| 全参数微调 | 8B | 64GB | 16GB | 96GB |
| LoRA (r=16) | 20M | 160MB | 40MB | ~200MB |

**性能对比**：
- 全参数微调：基准性能
- LoRA (r=16)：性能损失5-10%
- QLoRA (4-bit)：性能损失10-15%

**面试应用场景**：
> **Q：LoRA的秩r如何选择？**
>
> **A**：
> - r=8：参数最少，性能损失较大（10-15%）
> - r=16：平衡点，性能损失5-10%（推荐）
> - r=32：参数较多，性能损失2-5%
> - r=64：接近全参数，性能损失<2%
> 选择依据：任务复杂度、数据量、计算预算

#### 2.2 LoRA的实现细节

**关键配置**：
```yaml
lora_rank: 16          # 秩
lora_alpha: 32         # 缩放因子
lora_target: all       # 应用到所有层
```

**为什么lora_alpha=32？**
- 通常lora_alpha = 2 × lora_rank
- 缩放因子 = alpha / rank = 2
- 这个缩放因子可以平衡LoRA和原始权重的影响

**面试应用场景**：
> **Q：LoRA的alpha参数有什么作用？**
>
> **A**：alpha是缩放因子，控制LoRA更新的幅度：
> - 实际更新 = (alpha/rank) × BA
> - alpha越大，LoRA影响越大
> - 通常设置alpha = 2 × rank
> - 太小：LoRA效果弱，需要更多训练
> - 太大：可能破坏原始权重

### 3. RL训练的资源优化

#### 3.1 批量大小的权衡

**官方配置**：
- train_prompt_bsz=256
- n_resp_per_prompt=8
- 每步生成256×8=2048条轨迹

**我们的配置**：
- train_prompt_bsz=64
- n_resp_per_prompt=4
- 每步生成64×4=256条轨迹

**影响**：
- 组统计更 noisy（样本少）
- 方差更大
- 但仍然可以训练

**面试应用场景**：
> **Q：RL训练中batch size太小会有什么问题？**
>
> **A**：
> 1. **组统计不稳定**：GRPO依赖组内比较，样本少则统计不可靠
> 2. **方差大**：梯度估计方差大，训练不稳定
> 3. **收敛慢**：需要更多steps才能收敛
> 解决方案：
> - 增加gradient accumulation
> - 使用更稳定的优化器
> - 增加KL正则化

#### 3.2 序列长度的权衡

**官方配置**：
- max_prompt_length=4096
- max_response_length=70000
- 总长度：74000 tokens

**我们的配置**：
- max_prompt_length=2048
- max_response_length=16384
- 总长度：18432 tokens

**影响**：
- 无法处理需要很长推理链的任务
- 工具调用次数受限
- 但核心能力保留

**面试应用场景**：
> **Q：如何决定序列长度？**
>
> **A**：
> 1. **分析任务需求**：统计训练数据的序列长度分布
> 2. **设置分位点**：通常取95%分位点
> 3. **考虑显存限制**：序列长度每翻倍，显存增加约4×
> 4. **权衡性能损失**：太短会截断重要信息，太长会OOM
> 我们选择16384是因为：
> - 覆盖80%的训练样本
> - 显存限制在22GB以内
> - 仍然可以完成多轮工具调用

### 4. 工程落地的关键细节

#### 4.1 CPU Offload的性能开销

**开销来源**：
1. **数据传输**：PCIe带宽限制（约32GB/s）
2. **同步等待**：CPU和GPU之间的同步
3. **内存管理**：CPU内存的分配和释放

**实际测量**：
- 无Offload：100%速度
- 优化器Offload：70%速度（30%开销）
- 参数+优化器Offload：50%速度（50%开销）

**面试应用场景**：
> **Q：CPU Offload的性能开销是多少？如何优化？**
>
> **A**：
> - 优化器Offload：约30%开销
> - 参数+优化器Offload：约50%开销
> 优化方法：
> 1. **异步传输**：重叠计算和数据传输
> 2. **Pin Memory**：使用锁页内存加速传输
> 3. **预取**：提前加载需要的参数
> 4. **选择性Offload**：只Offload不常用的参数

#### 4.2 DeepSpeed ZeRO-3的通信开销

**通信模式**：
- AllGather：收集所有GPU的参数分片
- ReduceScatter：收集所有GPU的梯度分片

**通信量**：
- 每层：2 × 参数量 × (N-1)/N
- N=8时：约1.75 × 参数量

**优化方法**：
- 梯度累积：减少通信频率
- 通信重叠：重叠计算和通信
- 压缩通信：使用FP16/BF16通信

**面试应用场景**：
> **Q：DeepSpeed ZeRO-3的通信开销如何优化？**
>
> **A**：
> 1. **梯度累积**：多次前向/反向后再通信
> 2. **通信重叠**：在计算的同时进行通信
> 3. **压缩通信**：使用低精度通信
> 4. **分层通信**：节点内用NVLink，节点间用InfiniBand
> 5. **异步通信**：使用异步AllReduce

#### 4.3 训练稳定性

**常见问题**：
1. **梯度爆炸**：loss突然上升
2. **梯度消失**：loss不下降
3. **NaN/Inf**：数值溢出

**解决方案**：
```python
# 1. 梯度裁剪
max_grad_norm=1.0

# 2. 学习率warmup
warmup_ratio=0.1

# 3. 混合精度训练
bf16=True  # 比fp16更稳定

# 4. 损失缩放
loss_scale=0  # 自动缩放

# 5. 检查点恢复
resume_from_checkpoint=True
```

**面试应用场景**：
> **Q：训练过程中loss突然上升怎么办？**
>
> **A**：
> 1. **立即检查**：梯度是否爆炸（grad_norm）
> 2. **暂停训练**：保存当前checkpoint
> 3. **分析原因**：
>    - 数据问题？检查数据质量
>    - 学习率问题？降低学习率
>    - 模型问题？检查模型参数
> 4. **尝试恢复**：从上一个好的checkpoint恢复
> 5. **调整配置**：增加梯度裁剪、降低学习率

### 5. 数据工程的实践细节

#### 5.1 数据质量检查

**检查清单**：
```python
def check_data_quality(data):
    # 1. 格式检查
    assert "conversations" in data
    assert "images" in data
    
    # 2. 内容检查
    for conv in data["conversations"]:
        assert conv["from"] in ["human", "gpt", "observation"]
        assert len(conv["value"]) > 0
    
    # 3. 图像检查
    for img_path in data["images"]:
        assert os.path.exists(img_path)
        img = Image.open(img_path)
        assert img.size[0] >= 32 and img.size[1] >= 32
    
    # 4. 一致性检查
    # 检查对话轮次是否交替
    for i in range(len(data["conversations"]) - 1):
        if data["conversations"][i]["from"] == "human":
            assert data["conversations"][i+1]["from"] == "gpt"
```

**面试应用场景**：
> **Q：如何保证训练数据的质量？**
>
> **A**：
> 1. **格式检查**：确保数据格式正确
> 2. **内容检查**：确保内容非空、有意义
> 3. **图像检查**：确保图像可以正常加载
> 4. **一致性检查**：确保对话轮次交替
> 5. **统计检查**：检查长度分布、词频等
> 6. **抽样检查**：人工检查部分样本

#### 5.2 数据增强的实践

**增强类型**：
1. **图像增强**：模糊、降采样、透视畸变
2. **文本增强**：同义词替换、回译
3. **工具增强**：模拟工具失败

**增强比例**：
- 官方：10%增强子集
- 我们：可以增加到20%以提升鲁棒性

**面试应用场景**：
> **Q：数据增强有什么作用？如何选择增强比例？**
>
> **A**：
> 作用：
> 1. **提升鲁棒性**：让模型处理不完美输入
> 2. **增加多样性**：扩大训练分布
> 3. **防止过拟合**：减少对特定模式的依赖
>
> 比例选择：
> - 太小（<5%）：效果不明显
> - 太大（>30%）：可能引入噪声
> - 推荐：10-20%
> - 需要通过实验确定最佳比例

### 6. 评估与调试的实践

#### 6.1 评估指标的选择

**自动指标**：
- Pass@1：最终答案正确率
- 平均轮次：工具调用次数
- 致命轨迹比例：训练稳定性

**人工指标**：
- 答案质量：是否完整、准确
- 推理过程：是否合理、有逻辑
- 工具使用：是否高效、正确

**面试应用场景**：
> **Q：如何全面评估Agent的性能？**
>
> **A**：
> 1. **自动指标**：Pass@1、平均轮次、致命比例
> 2. **人工指标**：答案质量、推理过程、工具使用
> 3. **分层评估**：
>    - 简单任务：单轮问答
>    - 中等任务：多轮工具调用
>    - 困难任务：复杂推理链
> 4. **对比评估**：与基线模型对比
> 5. **消融评估**：验证每个组件的贡献

#### 6.2 调试策略

**调试流程**：
```
1. 复现问题
   ├── 最小化复现条件
   ├── 记录输入输出
   └── 确认一致性

2. 定位问题
   ├── 检查数据
   ├── 检查模型
   ├── 检查工具
   └── 检查配置

3. 分析原因
   ├── 是什么导致了问题？
   ├── 为什么会出现这种情况？
   └── 如何避免？

4. 解决问题
   ├── 尝试多种方案
   ├── 验证效果
   └── 记录经验
```

**面试应用场景**：
> **Q：遇到训练问题时如何调试？**
>
> **A**：
> 1. **复现问题**：确保问题可以稳定复现
> 2. **定位问题**：检查数据、模型、工具、配置
> 3. **分析原因**：找到根本原因
> 4. **解决问题**：尝试多种方案，验证效果
> 5. **记录经验**：避免类似问题再次发生
>
> 关键是要有系统性的调试方法，而不是随机尝试。

## 面试准备要点

### 1. 展示系统性思维

**不是**：我做了X，然后做了Y
**而是**：我分析了问题，发现瓶颈在A，所以用X解决；然后发现B，所以用Y解决

### 2. 展示深入理解

**不是**：我用LoRA减少了显存
**而是**：LoRA通过低秩分解减少可训练参数，秩r=16时参数量减少400倍，性能损失5-10%

### 3. 展示实践能力

**不是**：我配置了CPU Offload
**而是**：CPU Offload有30%性能开销，我通过异步传输和Pin Memory优化到20%

### 4. 展示问题解决能力

**不是**：训练很顺利
**而是**：训练过程中遇到梯度爆炸，我通过检查发现是数据问题，增加梯度裁剪后解决

### 5. 展示业务理解

**不是**：模型效果很好
**而是**：模型在简单任务上准确率90%，但在复杂任务上只有60%，需要针对复杂场景优化
